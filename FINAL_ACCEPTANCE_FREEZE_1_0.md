# NOVA 1.0 最终验收封板清单

本文档用于冻结 `NOVA 1.0` 的最终客户视角验收链、验收证据、通过标准与封板判定。

目标：

- 把前面所有封板项收成一次可执行、可判定、可留档的最终验收流程
- 避免 1.0 在最后阶段继续变成“永远差一点”的项目
- 给产品、实施、运维、测试一个统一的 1.0 通过标准

---

## 1. 最终验收定义

`NOVA 1.0` 最终验收的定义是：

> 按客户视角完整走通一条从启动、配置、登录、控制面初始化、配置发布、运行时验证到历史与审计回查的闭环，并留下可复核证据。

最终验收不是：

- 单个接口通过
- 单次本地调试成功
- 单个模块看起来可用

最终验收必须证明：

- 产品能被交付
- 产品能被使用
- 产品能被验证

---

## 2. 最终验收前置条件

只有下面全部满足，才允许开始最终验收：

1. 产品边界已冻结
2. 控制平面边界已冻结
3. Config Revision 边界已冻结
4. Runtime 边界已冻结
5. AI 质量边界已冻结
6. Windows EXE 交付边界已冻结
7. 部署与恢复边界已冻结

如果以上任一项未冻结，不进入最终验收。

---

## 3. 验收环境冻结

最终验收允许的环境路径只保留两种：

### 3.1 Windows EXE 路径

用于：

- 单机交付
- 演示
- 快速客户验收

交付物：

- `dist/NOVA/NOVA.exe`

### 3.2 服务端部署路径

用于：

- 私有化环境
- Compose / K8s 验收

交付物：

- `docker-compose.yml`
- `deploy/k8s/nova-deployment.yaml`

最终验收不再接受“第三种临时环境”作为正式通过依据。

---

## 4. 标准客户验收链

1.0 最终验收链固定为以下步骤：

### 4.1 启动

1. 启动 `NOVA.exe` 或服务端部署
2. 确认工作台或服务正常可达

### 4.2 检查基础状态

1. 确认 `/health` 正常
2. 确认 Studio 可进入
3. 确认 Config 页可读取配置

### 4.3 初始化控制平面

1. 创建 tenant
2. 创建 role
3. 创建 permission
4. 绑定 role-permission
5. 创建 user
6. 绑定 user-role

### 4.4 启用认证并登录

1. 开启 `auth.enabled`
2. 必要时重启
3. 使用创建的用户登录
4. 通过 `/api/auth/me` 或 Studio 验证用户上下文

### 4.5 配置治理

1. 创建 config revision
2. publish revision
3. rollback revision
4. 验证 revision 状态变化正确

### 4.6 运行时验证

1. 触发一条 runtime 事件
2. 查看 Dashboard 状态变化
3. 查看 Events
4. 查看 history
5. 查看 audit

### 4.7 收尾验证

1. 确认 control-plane 写操作均有审计
2. 确认 tenant scope 生效
3. 确认配置保存与重载链可解释

---

## 5. 必须留档的验收证据

最终验收必须至少保留以下证据：

### 5.1 启动证据

- EXE 启动截图或说明
- `/health` 响应

### 5.2 登录证据

- 登录成功说明
- `/api/auth/me` 返回结果或 Studio 当前用户上下文截图

### 5.3 控制平面证据

- tenant 创建结果
- role / permission / user 创建结果
- role-permission / user-role 绑定结果

### 5.4 配置治理证据

- revision create
- revision publish
- revision rollback

### 5.5 运行时证据

- Dashboard 状态
- Events
- history 查询结果
- audit 查询结果

最终验收必须“有证据”，不能只靠口头说明。

---

## 6. 通过标准

只有下面全部满足，最终验收才算通过：

1. 产品能启动
2. 工作台可访问
3. 配置可读取与保存
4. 控制平面最小初始化链可完成
5. 登录与当前用户上下文可验证
6. tenant scope 生效
7. config revision create/publish/rollback 可完成
8. runtime 事件链可观测
9. history / audit 可回查
10. 验收证据完整

---

## 7. 不通过条件

出现以下任一情况，最终验收直接判定不通过：

- EXE 或服务端无法稳定启动
- 工作台无法访问
- `/api/auth/me` 不可靠
- tenant scope 不生效
- revision 状态机不可靠
- history / audit 无法回查
- 文档步骤与实际产品行为不一致
- 需要大量人工临时补救才能跑通流程

---

## 8. 角色分工冻结

最终验收建议角色如下：

### 8.1 产品/交付负责人

负责：

- 确认是否满足 1.0 范围
- 最终签字

### 8.2 实施/测试人员

负责：

- 按手册逐步执行
- 记录证据

### 8.3 工程负责人

负责：

- 解释异常
- 确认是否属于阻断项

---

## 9. 最终 Go / No-Go

### Go

允许宣布 `NOVA 1.0` 封板完成，当且仅当：

- 8 个封板项都已整理完成
- 客户视角验收链已完整跑通
- 验收证据齐全
- 当前交付物与文档行为一致

### No-Go

以下任一情况成立，都不能宣布 1.0 完成：

- 还有未冻结的核心边界
- 验收链仍依赖隐性操作
- 文档不能直接指导交付
- 交付物与源码/文档严重不一致

---

## 10. 封板结论模板

建议最终结论统一使用以下模板：

### 通过

> NOVA 1.0 已按冻结边界完成最终验收。  
> 当前交付物、控制平面、配置治理、运行时、EXE 工作台、部署链与文档一致，可进入正式交付阶段。

### 不通过

> NOVA 1.0 暂未通过最终验收。  
> 阻断项为：`<列出阻断项>`。  
> 在阻断项修复并完成复验前，不进入正式交付。

---

## 11. 封板后的规则

一旦宣布 1.0 封板：

- 不再继续往 1.0 分支加大功能
- 新需求进入 1.1 或后续版本
- 1.0 只接受：
  - 阻断性修复
  - 交付性修复
  - 验收性修复

这条规则用于防止 1.0 封板后再次失控。
