# NOVA 1.1 路线图

本文档定义 `NOVA 1.1` 的产品方向、优先级、阶段目标与边界。

1.1 的前提是：

- `NOVA 1.0` 已完成封板
- 已有可交付 EXE / Compose / K8s 路径
- 已完成一次真实客户视角验收链

1.1 不再解决“能不能交付”的问题，重点转向：

- 配置治理做硬
- AI 质量做稳
- Studio 做强
- 服务端交付做厚
- 主平台链路做深

---

## 1. 1.1 产品目标

`NOVA 1.1` 的目标不是扩张成大而全平台，而是把 1.0 雏形成品升级成：

> 一个更稳定、更易运营、更易治理、更适合企业私有化落地的互动数字人平台版本。

1.1 核心目标：

1. 配置版本治理从“可用”升级到“可信”
2. AI 质量从“可运行”升级到“可回归”
3. Studio 从“最小后台工作台”升级到“可操作管理台”
4. 服务端部署从“可部署”升级到“可维护”
5. 主平台链路从“可演示”升级到“可长期运行”

---

## 2. 1.1 路线原则

1.1 必须遵守以下原则：

### 2.1 不追求大扩张

不在 1.1 同时追求：

- 全平台接入
- 多集群 SaaS
- 超复杂自治 Agent 系统

### 2.2 只做真实痛点

1.1 任务必须来自：

- 1.0 最终验收暴露的问题
- 交付与运维中的重复痛点
- 控制面与 runtime 的真实使用摩擦

### 2.3 每项改动都要能回归

1.1 不能继续靠“看起来更好”推进。

必须有：

- 文档
- 测试
- 证据

---

## 3. 1.1 五大主题

### 3.1 Theme A：Config Governance

目标：

- 把 `config revision` 真正做成 effective config 治理能力

1.1 要做：

- 为同一资源建立明确的 `effective revision` 读取链
- 明确 `published` 唯一性在数据库层或治理层的硬约束
- 让 runtime 读取“当前有效版本”，而不是仅仅支持 create/publish/rollback
- 给 revision 增加更明确的差异和状态展示

1.1 不做：

- 复杂审批平台
- 灰度流量编排系统

成功标准：

- runtime 能稳定消费当前有效 revision
- Studio/API 能明确展示当前有效 revision

### 3.2 Theme B：AI Quality

目标：

- 把 AI 行为纳入固定回归

1.1 要做：

- 建立真实最小 eval 数据集
- 建自动回归入口
- 收口以下 blocker：
  - 人格修正双重应用
  - tool-call 累积鲁棒性
  - memory consolidation 资源竞争
  - NLU 路由利用继续深化
  - emotion 影响链继续稳定

1.1 不做：

- 新一轮大模型能力炫技
- 新实验性长链规划玩法

成功标准：

- prompt / routing / model 改动有固定回归结论
- AI 质量变化可量化

### 3.3 Theme C：Studio Workbench

目标：

- 把 Studio 从“工程工作台”升级到“产品管理台”

1.1 要做：

- 资源列表更可读
- 详情页更明确
- 错误提示更直接
- 当前用户 / tenant scope / 权限关系更清楚
- config revision、history、audit 的阅读体验提升

1.1 不做：

- 重写成大型前端工程
- 做复杂设计系统工程

成功标准：

- 实施/运营用户不需要依赖研发解释就能完成核心后台操作

### 3.4 Theme D：Deployment & Ops

目标：

- 把服务端交付链从“能装”升级到“好维护”

1.1 要做：

- migration 策略更正式
- Postgres owner / grant / init 流程固化
- backup / restore 步骤进一步标准化
- 服务端验收脚本进一步模板化
- 健康检查与运维手册更稳

1.1 不做：

- 多云统一运维平台
- 复杂灾备自动化体系

成功标准：

- 新环境部署不需要大量临时修权限
- 运维问题可标准化处理

### 3.5 Theme E：Primary Platform Depth

目标：

- 继续把主平台链路做深，而不是扩平台数

建议 1.1 继续坚持：

- 只选一个主平台做深

推荐顺序：

1. Bilibili
2. Douyin webhook 验收链

1.1 要做：

- 主平台链路更稳定
- 真实事件输入更可靠
- 平台鉴权、消息映射、幂等和观测更稳

1.1 不做：

- 同时深挖所有平台

成功标准：

- 一个主平台可持续跑，不只是做验收注入

---

## 4. 1.1 分阶段计划

### Phase 1：治理与稳定

优先做：

- effective config 读取链
- published 唯一性治理
- Postgres owner / grant / migration 固化
- AI blocker 修复第一轮

### Phase 2：回归与工作台

优先做：

- 最小 AI eval 集
- Studio 资源阅读与操作体验增强
- revision/history/audit 视图增强

### Phase 3：主平台深挖

优先做：

- 主平台真实输入链路固化
- 主平台运行时监控和历史回查优化

---

## 5. 1.1 交付物建议

1.1 建议交付物：

- 更新后的 EXE 成品
- 更新后的 Compose/K8s 交付链
- AI eval 集与回归说明
- 更新后的 Studio 管理台
- 运维与迁移文档增强版

---

## 6. 1.1 不做清单

明确不进入 1.1 的内容：

- 新平台大扩张
- 通用插件市场
- 多租户 SaaS 商业系统
- 原生大型桌面框架重写
- 大规模 Agent 自治编排重构

---

## 7. 1.1 成功标准

如果下面条件成立，可以认为 1.1 成功：

1. effective config 链成立
2. AI 回归集建立并被使用
3. Studio 对非研发用户更友好
4. 服务端部署权限/迁移问题显著减少
5. 主平台链路可稳定跑通

---

## 8. 1.1 风险提醒

1.1 最大风险不是技术难度，而是再次发散。

必须避免：

- 看到一个问题就再开一个平台方向
- 看到一个机会就再开一个大模块
- 没有验收标准就开始重构

1.1 的核心纪律：

> 每一项工作都必须增强交付稳定性、治理能力或实际使用体验。

如果做不到，就暂缓。
