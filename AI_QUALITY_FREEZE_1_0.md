# NOVA 1.0 AI 质量封板清单

本文档用于冻结 `NOVA 1.0` 的 AI 质量目标、评测范围、阻断缺陷、回归规则与封板标准。

目标：

- 把 AI 能力从“能跑”收成“可测、可比、可验收”
- 防止 1.0 阶段继续凭感觉修改 prompt / routing / tool / memory
- 为 `Windows EXE 封板`、`部署与恢复封板`、`最终验收封板` 提供统一质量基线

---

## 1. 1.0 AI 质量定义

`NOVA 1.0` 的 AI 质量要求不是“最聪明”，而是：

- 回答稳定
- 人格一致
- 工具调用可靠
- 检索可解释
- 情绪联动真实可感
- 修改后可回归

1.0 不追求：

- 覆盖所有开放域问题
- 自治多 Agent 推理玩法
- 高度实验性的长链规划
- 每个场景都做到最优

1.0 的目标是：

> 在明确的互动数字人主场景中，输出质量稳定且可回归。

---

## 2. 1.0 AI 能力边界

1.0 AI 质量封板只围绕以下能力：

- semantic aggregation
- NLU intent classification
- emotion-aware routing
- memory-assisted response
- RAG retrieval
- tool calling
- safety-compatible output
- personality-consistent wording

1.0 封板前不再新增新的 AI 主能力类别。

---

## 3. 最小评测集冻结

1.0 AI 评测集冻结为 6 类：

1. `question`
2. `greeting`
3. `request/command`
4. `emotion response`
5. `rag answer`
6. `tool call`

建议每类样本量：

- 20 ~ 30 条

最小要求：

- 每类至少有固定样本
- 每次改动后跑同一套集
- 结果能前后比较

---

## 4. 质量判定维度

每条样本至少按以下维度判断：

### 4.1 相关性

- 是否真正回应了用户输入
- 是否偏题

### 4.2 完整性

- 是否给出足够可用的回答
- 是否无故中断

### 4.3 人格一致性

- 语气是否符合角色卡
- 是否出现明显人格漂移

### 4.4 工具可靠性

- 该调用工具时是否调用
- 调用后是否能继续完成自然语言输出
- 是否出现静默失败

### 4.5 检索质量

- 该走 RAG 时是否走 RAG
- 是否正确利用检索上下文
- 是否明显编造检索结论

### 4.6 情绪联动

- 情绪是否真实影响回应风格
- 是否仍只是“参数调温器”

### 4.7 安全性

- 是否明显绕过 safety
- 是否输出不该输出的内容

---

## 5. 1.0 必须冻结的实现原则

### 5.1 语义聚合

1.0 必须固定为：

- embedding 语义聚类路径

1.0 禁止回退为：

- TF-IDF
- 中文单字切分相似度聚类

### 5.2 NLU

1.0 必须继续保留：

- NLU 对 routing 的真实影响

NLU 不允许退回成：

- 只写日志
- 只做 metadata 注释

### 5.3 Emotion

1.0 必须至少影响：

- routing style
- prompt tone
- generation prosody /表达风格

Emotion 不允许退回成：

- 只调整 temperature

### 5.4 Tool calling

1.0 必须保证：

- 工具轮次结束后文本输出连续
- 工具失败有可见错误，而不是静默丢失

### 5.5 Memory

1.0 必须明确：

- memory consolidation 不能无限制与主响应争抢关键推理资源

---

## 6. 1.0 阻断缺陷（Blockers）

以下问题在 1.0 中属于 AI 质量阻断项，必须优先收口：

### 6.1 人格修正双重应用

问题：

- 句子级修正一次
- 全文级再次修正一次

风险：

- 流式输出与最终输出不一致
- persona 表现失真

### 6.2 流式 tool-call 累积鲁棒性

问题：

- 多工具/索引不连续场景下累积逻辑不够稳

风险：

- 工具调用静默失败
- 输出中断或与工具结果断链

### 6.3 memory consolidation 资源竞争

问题：

- consolidation 与主响应可能竞争同一推理资源

风险：

- 响应时延抖动
- 直播互动体验下降

### 6.4 NLU 利用深度不足

问题：

- 虽已进入 routing，但利用深度仍需继续校准

风险：

- 问答、命令、检索、闲聊之间策略仍可能不够分明

### 6.5 Emotion 影响面不够广

问题：

- emotion 已进入 routing / prompt / 部分表达链，但仍需进一步固化

风险：

- 情绪表现不稳定
- 用户感知不到明显差异

---

## 7. 回归规则冻结

从本清单起，以下改动都必须跑最小 AI 回归集：

- 改模型
- 改 prompt
- 改 routing
- 改 tool calling
- 改 retrieval
- 改 memory 策略
- 改 emotion 联动逻辑

未经回归集验证，不允许宣称 AI 质量提升。

---

## 8. 结果记录要求

每轮 AI 质量改动至少要留下：

- 改动点
- 样本类别
- 预期变化
- 回归结果
- 结论

目的不是做科研论文，而是保证：

- 质量变化可回看
- 问题可复现
- 回滚有依据

---

## 9. Studio / 控制面要求

1.0 不要求 Studio 做完整 AI 评测平台，但必须能支持最小验证：

- 能看到 runtime history
- 能看到 audit
- 能看到配置状态
- 能辅助人工完成一次质量验收链

AI 评测结果在 1.0 可先通过文档/测试报告管理，不强求 Studio 可视化评测中心。

---

## 10. 1.0 AI 验收标准

只有下面全部成立，`AI 质量封板` 才算通过：

1. 6 类最小评测集已冻结
2. 语义聚合保持 embedding 路径
3. NLU 对 routing 的真实影响稳定存在
4. emotion 对风格/语气的真实影响稳定存在
5. tool calling 不再出现明显静默失败链
6. 人格一致性没有明显回退
7. 关键已知 blocker 已至少被记录并进入收口清单

---

## 11. 1.0 剩余收口重点

AI 质量在 1.0 剩余最值得继续做的事情只保留：

1. 固定最小 eval 集
2. 修 blocker
3. 稳定 routing
4. 稳定 persona
5. 稳定 tool calling

1.0 阶段不再扩张新的 AI 玩法。

---

## 12. Go / No-Go 判定

### Go

允许进入下一个封板项（`Windows EXE 封板`），当且仅当：

- AI 回归范围已冻结
- blocker 列表已明确
- routing / tool / persona 的质量底线已确定

### No-Go

以下任一情况成立，都不允许认为 AI 质量已封板：

- 仍然凭感觉改 prompt
- 没有固定评测集
- 语义聚合退回旧路径
- 工具链静默失败仍频繁存在
- 人格输出前后明显不一致

---

## 13. 下一步

AI 质量封板完成后，严格进入下一个封板项：

- `Windows EXE 封板`

下一阶段重点不再是“AI 再加点新能力”，而是把现有能力封进真正可交付、可操作、可演示的桌面成品。
